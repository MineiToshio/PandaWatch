# FRD-006: Slug Generation

**Version:** 1.0  
**Status:** Draft  
**Author:** Architecture session 2026-05-27  
**Related:** [BP-002](blueprints/BP-002-url-routing.md), [scripts/retrofit/README.md](../../scripts/retrofit/README.md)

---

## Overview

Every item in `data/items.jsonl` needs a `slug` field — a URL-safe, unique,
human-readable identifier used as the path segment in `/item/[slug]`.

The slug is generated by a Python script (`scripts/retrofit/generate_slugs.py`),
not by an LLM. It is deterministic, idempotent, and pipeline-friendly.

---

## Problem Statement

The existing Alpine.js app used hash routing (`#/volume/{url}`) — the URL itself
was used as the identifier. This is impractical for a real URL scheme:
- Raw URLs contain domains, query params, and special characters
- Not human-readable or SEO-friendly
- Breaks when a source URL changes (e.g., a retailer changes its product slug)

Deriving the slug on-the-fly at render time from `edition_key + volume` is tempting
but fragile: the computation could produce different results after a code change,
breaking existing indexed/bookmarked URLs. A stored `slug` field eliminates this risk.

---

## User Stories

- As a **developer**, I want every item to have a `slug` I can use directly in
  Next.js routes without any transformation at render time.
- As a **future DBA**, I want to add `slug TEXT UNIQUE NOT NULL` to the SQLite
  schema and it just works — no migration needed.
- As a **search engine**, I want URLs like `/item/berserk-darkhorse-deluxe-42` that
  are stable and meaningful.

---

## Functional Requirements

### FR-1: Slug derivation rules

Slugs are derived per cluster (not per row). All rows sharing a `cluster_key`
get the same `slug` value.

**Priority order:**

```
1. cluster_key = "isbn:{isbn13}"
   → slug = "isbn-{isbn13}"
   → example: isbn-9781506721910

2. edition_key + volume (numeric or alphanumeric)
   → slug = "{edition_key}-{formatted_volume}"
   → volume formatting: int(float(v)) for whole numbers, str(v) for decimals
   → examples:
        berserk-darkhorse-deluxe-42
        one-piece-viz-collector-100
        demon-slayer-shueisha-special-23
        cardcaptor-sakura-darkhorse-deluxe-1

3. edition_key + no volume (box sets, standalone artbooks)
   → slug = "{edition_key}"
   → examples:
        gon-norma-collector
        nier-kurokawa-artbook
        dragon-ball-20aniv-norma-boxset

4. no edition_key + has isbn (pre-standardization items with ISBN)
   → slug = "isbn-{isbn}"
   → (same as rule 1, isbn field used directly)

5. fallback (no edition_key, no isbn)
   → slug = "item-{sha1(canonical_url)[:12]}"
   → example: item-a3f9b12ce7d1
   → note: these are items not yet processed by /watch-standardize-catalog;
            their slug will stabilize after standardization runs.
```

### FR-2: Collision resolution

If two different clusters produce the same candidate slug (possible in edge cases
with very short series names or unusual edition combinations):

1. Detect collisions after computing all candidate slugs
2. For all clusters in a collision group, append `-b`, `-c`, etc. to their slug,
   sorted by `detected_at` ascending (oldest item keeps the "clean" slug)
3. Log a warning with the collision details so the issue can be investigated

Example:
- `one-piece-shueisha-special` (cluster A, oldest) → `one-piece-shueisha-special`
- `one-piece-shueisha-special` (cluster B, newer) → `one-piece-shueisha-special-b`

### FR-3: Idempotency

Once a slug is assigned, it must not change on subsequent runs **unless** the
underlying `edition_key` or `volume` changed (which would indicate a re-standardization).

The script checks: if `current_slug` is already set and the `edition_key` / `volume`
fields are unchanged, skip that item (no re-write). Only update when:
- `slug` is empty/null
- `edition_key` changed since last slug generation
- `volume` changed since last slug generation

This prevents URL rot for already-deployed/indexed pages.

### FR-4: Volume number formatting

| Input | Output |
|---|---|
| `"42"` | `"42"` |
| `"42.0"` | `"42"` (strip trailing .0) |
| `"1.5"` | `"1-5"` (replace `.` with `-`) |
| `"10-12"` | `"10-12"` (already slug-safe) |
| `"第42巻"` | `"42"` (strip JP markers) |
| `"Vol. 42"` | `"42"` (strip prefix) |

### FR-5: Character safety

All generated slugs must match `^[a-z0-9][a-z0-9-]*[a-z0-9]$`:
- Lowercase ASCII alphanumeric and hyphens only
- No leading or trailing hyphens
- No consecutive hyphens

The `edition_key` field already satisfies this constraint (it's generated by the
same slug rules in `/watch-standardize-catalog`). Volume number formatting (FR-4)
ensures the volume suffix is also safe.

### FR-6: Script interface

```
scripts/retrofit/generate_slugs.py [--dry-run] [--only-missing] [--verbose]

Options:
  --dry-run        Print changes without writing to items.jsonl
  --only-missing   Only process rows where slug is empty/null (fast incremental)
  --verbose        Print each slug assignment

Output:
  - Writes slug field to all rows in items.jsonl
  - Prints summary: N slugs generated, M skipped, K collisions resolved
  - Logs to logs/ with timestamp
```

### FR-7: Pipeline integration

The script runs as the **last step of the `/watch-standardize-catalog` skill** — after
items have been written to `items.jsonl` with their canonical fields, the skill
calls:

```bash
.venv/bin/python scripts/retrofit/generate_slugs.py --only-missing --verbose
```

This assigns slugs to newly standardized items. Items already standardized retain
their existing slug (`--only-missing` skips them).

The script does **not** run automatically in `scrape_delta.sh` or `scrape_full.sh`.
Slugs are only generated when `/watch-standardize-catalog` runs. See `scripts/retrofit/README.md`
for the full implementation details.

## Non-Functional Requirements

- **Performance:** Script completes in < 5s for 10k items.
- **Safety:** Script uses `backup_and_rotate()` before writing to items.jsonl.
- **Stability:** Slug for any given item must be identical across multiple runs
  on the same corpus (barring collision resolution changes).

---

## Out of Scope

- LLM-based slug generation (simple deterministic logic is sufficient).
- Slug aliases or redirects (if a slug changes, old URLs break — avoid by
  making slugs stable per FR-3).

---

## Acceptance Criteria

- [ ] Every item in items.jsonl has a non-empty `slug` after running the script
- [ ] All slugs match `^[a-z0-9][a-z0-9-]*[a-z0-9]$`
- [ ] All slugs are unique across the corpus (no duplicates)
- [ ] Re-running the script on already-slugged items produces no changes
- [ ] `--dry-run` prints changes without modifying items.jsonl
- [ ] `--only-missing` only processes items with empty slug (fast path)
- [ ] Collision warning is printed for any detected collisions
- [ ] Backup created before write

---

## Dependencies
