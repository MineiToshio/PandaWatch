¿Qué es?

Un script en Python que recorre 82 fuentes (editoriales oficiales, tiendas, medios y redes) en cinco idiomas (español, inglés, francés, italiano y japonés) y busca señales de ediciones coleccionables: tapa dura, limitadas, numeradas, boxsets, variant covers, artbooks, 限定版, 特装版, etc.

Cada día genera un reporte Markdown con lo nuevo y lo cambiado respecto al día anterior, para que no tengas que pasar por 82 webs a mano.

Instalación

Solo si todavía no tenés el entorno listo:

python3 -m venv .venv
.venv/bin/pip install -r requirements.txt

Para correr los tests también:

.venv/bin/pip install -r requirements-dev.txt

Tu primer run

Lo más seguro: corré primero un dry run para ver qué encuentra sin escribir nada en disco.

.venv/bin/python manga_watch.py --dry-run

Esto va a:

  • Pegar las 82 fuentes (con 1.5s de pausa entre cada una → tarda 3–5 min).
  • Mostrarte en consola qué candidatos detectó por fuente.
  • No escribir state.json, items.jsonl ni reporte.

Cuando estés conforme, sin --dry-run ya escribe el reporte en reports/YYYY-MM-DD.md.

Todos los flags

Estos son todos los flags del CLI, agrupados por para qué sirven. Los marcados con NUEVO son de la última iteración (Fase 1).

Selección de fuentes

--sources <ruta>
Default: sources.yml

Path al YAML con la lista de fuentes. Solo lo tocarías si tenés varios YAMLs (por ejemplo sources-test.yml para probar fuentes nuevas sin romper el flujo).

--source-classes official,retailer
Default: vacío (todas las clases)

Filtra por clase de fuente. Valores válidos:

  • official — editoriales oficiales (Panini, Norma, Ivrea, VIZ…).
  • retailer — tiendas con exclusivas (Kinokuniya, Amazon JP…).
  • trusted_media — medios especializados (ListadoManga, Manga-news…).
  • social — redes/blogs (menos estables, más ruido).

Ejemplo típico para reducir ruido:

--source-classes official,retailer
--countries "España,Francia"
Default: todos

Filtra por país, exactamente como aparece en el YAML (con tildes y todo). Útil si solo te interesa una región.

--countries Japón
--only-source "MX - Panini Manga México" NUEVO
Default: vacío

Procesa una sola fuente por nombre exacto. Si te equivocás escribiendo el nombre, el script lista los nombres disponibles para que copies el correcto.

Es la forma rápida de iterar sobre una fuente puntual sin esperar a las 82.

--include-disabled
Default: off

Por default las fuentes con enabled: false en el YAML se saltan. Con este flag se incluyen igual. Útil para reactivar temporalmente una fuente sin tocar el YAML.

--list-sources
Default: off

Imprime las fuentes (después de aplicar los filtros de arriba) y termina sin hacer fetch. Sirve para chequear "¿qué se va a correr?" antes de lanzar el run completo.

Filtros de detección

--min-score 30
Default: 30

Score mínimo para que un candidato aparezca en el reporte.

  • Bajarlo (--min-score 20) muestra más cosas, incluyendo señales débiles.
  • Subirlo (--min-score 60) deja solo hallazgos fuertes.
Ojo: esto NO filtra lo que se guarda en state.json ni los logs en consola. Solo afecta lo que termina en el Markdown.
--fuzzy-keywords NUEVO
Default: off

Por defecto el script busca frases exactas ("edición especial", "limited edition"). Con este flag, además busca palabras sueltas de cada regla — útil para variantes como "tomo especial", "manga especial", etc.

Ejemplo: la regla "edición especial" → 40 pts

  • Si el texto dice "edición especial": suma 40 pts.
  • Si el texto dice "tomo especial": con fuzzy off suma 0, con fuzzy on suma 13 pts (40÷3).
  • Si el texto dice solo "edición" (sin nada más): suma 0, porque "edición" es stopword.

El script extrae automáticamente las palabras "fuertes" de cada regla, ignorando stopwords ("edición", "de", "la", "the", "of", "manga", "vol", "tomo"…).

Trade-off: captura más ediciones que se llaman distinto, pero también genera algo más de ruido. Los matches fuzzy se marcan en el reporte como [fuzzy:especial] para distinguirlos.

--fuzzy-divisor 3 NUEVO
Default: 3

Cuánto se divide el score original cuando matchea solo una palabra suelta (no la frase completa). Mayor divisor = score más bajo para matches fuzzy = menos ruido.

  • --fuzzy-divisor 2: matches fuzzy valen 50% del score original (más permisivo)
  • --fuzzy-divisor 3: 33% (default, balanceado)
  • --fuzzy-divisor 5: 20% (más conservador)
--max-age-days 30 NUEVO
Default: 30 días

Solo se aplica a fuentes RSS: descarta entradas con fecha publicada más vieja que N días. Con 0 se desactiva el filtro.

Si una entrada tiene una fecha rara que el script no puede parsear, se deja pasar (no penalizamos a feeds con formatos no estándar).

--include-seen
Default: off

Por default el reporte solo muestra hallazgos new o changed. Con este flag también incluye los seen (los que ya viste antes).

Útil cuando querés ver "todo lo que matchea hoy", no solo lo nuevo. También útil para auditar qué está detectando en general.

--max-items-per-source 80
Default: 80

Máximo de candidatos a extraer por fuente. Solo se nota en páginas con muchísimos productos. Bajarlo acelera el procesamiento; subirlo no suele cambiar nada.

Comportamiento HTTP

--sleep-seconds 1.5
Default: 1.5 segundos

Pausa entre fuente y fuente.

  • Bajarlo a 0.5 acelera el run pero podés llamar la atención de Cloudflare.
  • Subirlo a 3 es más educado con los servidores.
--connect-timeout 10
Default: 10 segundos

Cuánto espera para establecer la conexión TCP. Si una fuente no responde en ese tiempo, se cancela el intento.

--read-timeout 30
Default: 30 segundos

Cuánto espera para terminar de leer la respuesta una vez conectado. Si una página tarda más en cargar, se cancela.

--user-agent "..."
Default: manga-watch-personal/0.2 (+personal-use)

El User-Agent que mandamos en cada request. Algunas tiendas filtran UAs raros. Si una fuente te bloquea con 403, probá uno de Chrome.

--respect-robots
Default: off FASE 2

Si está presente, lee robots.txt de cada dominio y saltea las URLs que no permite.

Está off por default. En la Fase 2 vamos a invertir esto: debería ser opt-out.

Output y modo de ejecución

--data-dir data
Default: data/

Dónde se escriben state.json (estado para diff entre días) y items.jsonl (historial append-only).

--reports-dir reports
Default: reports/

Dónde se escribe el reporte Markdown del día (YYYY-MM-DD.md).

--dry-run
Default: off

Corre todo el pipeline pero no escribe nada: ni state, ni JSONL, ni reporte, ni Telegram. Ideal para probar cambios sin contaminar el estado.

--send-telegram
Default: off

Después de generar el reporte, manda un resumen por Telegram. Requiere TELEGRAM_BOT_TOKEN y TELEGRAM_CHAT_ID en .env.

Se ignora si --dry-run está activo.

--enable-js NUEVO
Default: off

Activa el rendering con Playwright (Chromium headless) para fuentes marcadas con kind: "js" en el YAML. Off por default para que el script no falle si no tenés Playwright instalado.

Requiere instalación opt-in:

.venv/bin/pip install -r requirements-playwright.txt
.venv/bin/python -m playwright install chromium
--diagnostic NUEVO
Default: off

Activa el modo diagnóstico: durante el run captura por fuente HTTP status, content-type, tiempo de fetch, tamaño del HTML, # de anchors, método de extracción usado, # de cards detectados, # de candidatos con señales, top títulos y señales, y errores.

Al final escribe:

  • logs/diagnostic-<timestamp>.json — estructurado, machine-readable.
  • logs/diagnostic-<timestamp>.md — legible, agrupado por estado.
  • logs/raw/<slug>.html — primeros 80 KB del HTML para fuentes problemáticas (sin candidatos, vacías, JS, errores).

Ideal para auditar qué fuentes funcionan bien, cuáles necesitan selectores manuales, cuáles requieren Playwright, etc.

--log-dir logs NUEVO
Default: logs/

Directorio donde escribir los outputs del modo diagnóstico. Solo se usa cuando --diagnostic está activo.

--list-empty-sources NUEVO
Default: off

Al final del run imprime las fuentes que se detectaron como vacías o JS-rendered. El detector marca como problemáticas las páginas con:

  • HTML muy corto (menos de 5000 chars).
  • SPA shells vacíos: <div id="root">, #app, #__next…
  • Menos de 5 enlaces con texto significativo.

Sirve para decidir cuáles van a necesitar Playwright en una próxima fase.

Modo diagnóstico — auditar qué funciona y qué no NUEVO

Cuando quieras saber por qué una fuente no devuelve nada, o qué páginas son JS-rendered, o cuáles tardan demasiado, corré con --diagnostic:

.venv/bin/python manga_watch.py --diagnostic --dry-run

Esto genera tres tipos de archivos en logs/:

1. logs/diagnostic-<timestamp>.md — el reporte legible

Agrupa las fuentes por estado para que veas de un golpe qué anda:

EstadoSignifica
✅ OKLa fuente devolvió candidatos con señales.
⚠️ OK pero sin candidatosEl HTML cargó bien pero no detectó cards / no hay matches de keywords. Suele necesitar selectores manuales en el YAML.
🚫 Vacío / muy cortoHTML < 5000 chars.
🧩 JS-renderedSPA shell vacío. Necesita Playwright.
🔗 Sin enlacesMenos de 5 anchors con texto significativo.
❌ HTTP error404, 403, 503, etc.
🌐 Error de redTimeout, DNS, SSL, etc.
🤖 RobotsBloqueada por robots.txt.

Por cada fuente vas a ver:

  • HTTP status + content-type + tiempo de fetch.
  • Tamaño del HTML y cantidad de anchors.
  • Método de extracción usado (yaml-selectors, clusters, rss, none).
  • Cuántos "cards" se descartaron y por qué (sin anchor, descripción muy corta/larga, URL duplicada, sin señales).
  • Top 5 títulos con score y top señales.
  • Mensaje de error si lo hubo.

2. logs/diagnostic-<timestamp>.json — para análisis programático

Mismo contenido pero estructurado. Útil si querés hacer queries con jq o cargarlo en un script para comparar runs.

# ejemplos con jq
jq '.summary.by_status' logs/diagnostic-*.json
jq '.sources[] | select(.status == "no-candidates") | .name' logs/diagnostic-*.json
jq '.sources[] | select(.fetch_ms > 5000) | {name, fetch_ms}' logs/diagnostic-*.json

3. logs/raw/<slug>.html — HTML crudo de fuentes problemáticas

Para cada fuente que terminó como no-candidates / vacía / JS / error, se guarda un dump de los primeros 80 KB del HTML que devolvió el servidor. Esto te deja (o me deja, si compartís los archivos) abrirlo y ver exactamente qué hay que cambiar:

  • ¿Hay productos pero con un layout raro? → agregar selectors al YAML.
  • ¿Hay <div id="root"> vacío? → necesita Playwright.
  • ¿Hay un challenge de Cloudflare? → ajustar User-Agent o saltear.
Recomendado para tu primer audit: 
.venv/bin/python manga_watch.py \
  --diagnostic \
  --dry-run \
  --source-classes official,retailer \
  --sleep-seconds 0.5
Solo fuentes oficiales y retailers (las más accionables), sin pausas largas, sin escribir nada al state. Tarda ~1-2 minutos y deja todo lo necesario para revisar.

Fuentes JS (Playwright) NUEVO

Algunas tiendas modernas (Crunchyroll, Penguin Libros, Tiendanube, etc.) renderizan los productos con JavaScript. Devuelven un HTML casi vacío para requests. Para esas, usamos Playwright: un browser real headless que ejecuta el JS antes de leer el contenido.

1. Instalación (una sola vez)

.venv/bin/pip install -r requirements-playwright.txt
.venv/bin/python -m playwright install chromium

Esto baja Chromium (~92MB). Sin esta instalación, las fuentes kind: "js" se saltean con un mensaje claro — el resto del pipeline sigue funcionando normal.

2. Marcar fuentes en el YAML

En sources.yml, las fuentes que requieren JS llevan kind: "js":

- name: "AR - La Comiquería"
  source_class: "retailer"
  kind: "js"             # <-- requiere Playwright
  url: "https://www.lacomiqueria.com.ar/"

Las 14 fuentes JS ya detectadas y marcadas:

  • AR - La Comiquería
  • ES - Crunchyroll Noticias
  • ES - Distrito Manga
  • ES - Kibook Novedades
  • ES - Misión Tokyo (novedades)
  • ES - Misión Tokyo (lanzamientos)
  • FR - Meian
  • FR - Meian Plus Boutique
  • FR - Vega Dupuis
  • IT - J-Pop Manga
  • JP - Shueisha Comics Paper Releases
  • JP - Shogakukan Comics Calendar
  • SOCIAL - Manga Mogura Bluesky
  • US - Seven Seas Box Sets

3. Correr con JS habilitado

.venv/bin/python manga_watch.py --enable-js --dry-run

Las fuentes kind: "js" se renderizan con Chromium; las demás siguen usando requests normal. El browser se reutiliza entre fuentes (es lento abrirlo una sola vez, no 14).

Qué hace internamente

  1. Lanza Chromium headless con User-Agent realista de Chrome desktop.
  2. page.goto(url) con wait_until: domcontentloaded.
  3. Espera con wait_for_function hasta que aparezca algún <a> con texto ≥5 chars (máx. 8s).
  4. Hace scroll progresivo de toda la página para disparar lazy-load.
  5. Lee page.content() y lo pasa al mismo pipeline de extracción.
Tip de debugging: Si una fuente JS no funciona, corré sola con 
.venv/bin/python manga_watch.py --diagnostic --dry-run --enable-js \
  --only-source "ES - Crunchyroll Noticias"
El logs/raw/<slug>.html tendrá el HTML ya renderizado con todo el contenido visible.

Cómo leer el reporte

El reporte Markdown (reports/YYYY-MM-DD.md) tiene esta estructura:

SecciónQué contiene
ResumenConteo de hallazgos urgentes, interesantes, bajos y errores.
🔥 UrgentesScore ≥ 70. Señales fuertes: limitada, numerada, collector, boxset…
✅ InteresantesScore 35–69. Tapa dura, deluxe, hardcover, etc.
🎨 BajosScore < 35. Suelen ser artbooks simples o cosas a revisar.
Tabla rápidaUna fila por hallazgo con score, país, editorial, título y link.
Fuentes problemáticas NUEVOFuentes JS-rendered, vacías, con error HTTP, bloqueadas por robots.txt, etc. — agrupadas por categoría.
ErroresLista cruda de errores por fuente.

Cada hallazgo tiene:

  • Score: la suma de las señales detectadas (cap a 100).
  • Estado: new (nuevo), changed (cambió respecto a la última vez), seen (ya visto).
  • Tipo de producto NUEVO: manga / artbook / fanbook / guidebook / boxset / novel.
  • Precio NUEVO: si se detecta — multi-moneda (€, $, ¥, £, MXN). Vacío si no aparece.
  • Fecha de lanzamiento NUEVO: si se detecta — soporta ISO, dd/mm/yyyy, japonés (`2026年6月15日`), mes en EN/ES/FR/IT.
  • Imagen NUEVO: URL de la portada cuando la card tiene <img>. Se muestra inline en el Markdown.
  • Señales: las frases concretas que matchearon ("edición limitada", "hardcover", "限定版"…).
  • Fragmento: un extracto del bloque donde se detectó.

Si algún campo no se puede extraer, queda vacío — no es error. El sistema reporta lo que está disponible en la página.

Telegram (opcional)

Para recibir un resumen diario en Telegram:

  1. Hablá con @BotFather en Telegram y creá un bot. Te da un token.
  2. Mandale un mensaje cualquiera a tu bot.
  3. Abrí https://api.telegram.org/bot<TU_TOKEN>/getUpdates en el browser y copiá tu chat_id.
  4. Pegalos en .env:
TELEGRAM_BOT_TOKEN=123456:ABC-def...
TELEGRAM_CHAT_ID=987654321

Después corré con:

.venv/bin/python manga_watch.py --send-telegram

Correr los tests

Si tocaste lógica de extracción o querés verificar que todo sigue funcionando:

.venv/bin/python -m pip install -r requirements-dev.txt
.venv/bin/python -m pytest tests/ -v

Los tests cubren las funciones nuevas de Fase 1: detección de clusters de productos, detección de páginas JS-rendered, y parseo de fechas RSS.
